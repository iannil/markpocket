# ADR-0002：软实时广播 + Last-Write-Wins

- **状态**：Accepted
- **日期**：2026-06-30
- **相关**：取代 teable 的 share-db / op-log / op-builder 实时栈

## 背景

teable 的实时协作基于 share-db（OT 协议），含 op 持久化、op 回放、并发合并。这是 teable 后端第二大复杂度来源，且依赖一个独立维护的实时中间件。

markpocket 需要决定：实时协作做到什么粒度。

## 决策

采用**软实时广播 + Last-Write-Wins**：

- 单一 Next.js 进程内嵌 WebSocket 网关
- 每个 Base 一个频道，客户端订阅后接收 `ChangeEvent`
- 任何写操作在事务提交后，由服务端构造事件广播给同频道其他客户端
- 并发对同一 Cell 的写入：以**服务端到达时间戳**为序，后写覆盖前写（LWW），不合并、不分叉
- 客户端断线重连后**全量重拉**当前 Base 状态，不做 op 回放

## 后果

**正面**

- 不需要 op-log 持久化、不需要 OT 算法、不需要 share-db
- 实时逻辑可由小团队完全掌握与维护
- 服务端无状态化更容易（多实例时只需 Redis pubsub 转发）

**负面**

- 并发同字段编辑会出现"后写覆盖前写"，用户可能感知数据丢失 → UI 必须在检测到冲突时提示
- 断线重连成本高（全量重拉），不适合超长会话 + 频繁断网场景
- 无 op 历史（与"字段级历史"分离：历史是值快照，不是操作日志）

## 备选方案

- **OT（share-db 式）**：teable 路线。维护成本高，违背初衷。否决。
- **CRDT（Yjs / Automerge）**：合并体验好，但引入大型依赖与全新数据模型，且 Postgres 持久化 CRDT 文档型数据需额外设计。否决。
- **不做实时（轮询/刷新）**：最简但产品体验降级过多。否决。

## 反悔代价

中高。从 LWW 升级到 OT/CRDT 需要引入 op-log、客户端协作状态机、冲突解决算法，几乎是一次独立的子系统重写。该决策应在 v1 上线、获得真实并发数据后再评估是否升级。
