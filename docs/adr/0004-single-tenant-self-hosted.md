# ADR-0004：单租户自托管（不构建 SaaS 多租户）

- **状态**：Accepted
- **日期**：2026-06-30
- **相关**：取代 teable 的 organization / invitation / 计费相关 feature

## 背景

teable 设计为可 SaaS 化的产品，含：

- organization / workspace 层级
- 邀请邮件、成员管理
- 计费、用量限额（部分在闭源 hosted 版）
- 多租户隔离（基于应用层 tenant_id）

markpocket 当前未上线，需要决定部署与租户形态。

## 决策

采用**单租户自托管**：

- 一个 Docker Compose（web + postgres）服务**一个客户/团队**
- 数据模型中**不引入 tenant_id**：所有表默认属于部署实例
- 不做计费、邀请邮件、用量限额、SSO 强制
- better-auth 仅做本实例的本地账号管理（+ 可选 OIDC 给单一企业对接）

## 后果

**正面**

- 数据模型大幅简化（无 tenant_id 渗透、无 RLS）
- 部署最简（一个 Compose 文件，无控制面）
- 适合开源分发 / 私有交付

**负面**

- 服务多个客户需多实例部署（运维成本随客户数线性增长）
- 未来转 SaaS 需补 tenant 层：在所有业务表加 `tenant_id` + 加 RLS 或应用层过滤，工作量中等
- 不内置计费，商业化需在 v2 单独立项

## 备选方案

- **SaaS 多租户（v1 即做）**：未来商业化更顺，但当前复杂度收益不明显、且未上线无此压力。否决。
- **单租户优先 + 预留 tenant_id 字段**：折中方案，但预留字段会污染所有 schema 与查询，未上线前收益不抵成本。否决，待转 SaaS 时一次性补齐。
- **Cloud-native（k8s + Helm）**：运维更重，v1 客户规模用不上。否决，留作未来。

## 反悔代价

中。从单租户转多租户是侵入式重构（schema 加列 + 查询加过滤 + 鉴权改造），但因 v1 数据模型已尽量保持扁平，转换路径清晰。该决策应在出现"多客户共享同一部署"的真实商业需求时重新评估。
